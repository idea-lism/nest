# ANTLR g4 importer

`src/g4.c` is a small source-to-source importer from ANTLR `.g4` grammars to Nest grammar files.

## Mandatory implementation architecture

This importer is a bootstrapping exercise for Nest itself. Recognition of `.g4` syntax **must** be performed by a Nest grammar compiled by `nest`.

Required implementation shape:

1. Create and maintain a dedicated Nest grammar for ANTLR `.g4` input, for example `src/g4.nest` or another checked-in `.nest` file chosen by the build.
2. Add build rules that compile that Nest grammar with the Nest compiler and link the generated lexer/parser artifacts into the `g4` executable.
3. Implement `src/g4.c` only as the driver/converter around the generated parser: command-line handling, invoking the generated parser, walking the parse tree/AST, diagnostics, semantic conversion, and Nest output generation.

Forbidden implementation shapes:

- Do **not** hand-write a `.g4` lexer or parser in C.
- Do **not** implement recognition as a regex-only scanner.
- Do **not** use ANTLR, yacc/bison, tree-sitter, or any other parser generator for `.g4` syntax.
- Do **not** check in a C file whose tokenization/parsing logic recognizes `.g4` directly instead of calling the Nest-generated parser.

If the generated Nest parser cannot yet recognize a `.g4` construct, extend the dedicated `.nest` grammar first. The checked-in C importer may contain command-line handling, AST conversion, diagnostics, and Nest output generation, but recognition of `.g4` syntax must come from the Nest-generated lexer/parser artifacts.

The importer is intentionally conservative: it preserves ANTLR's high-level lexer/parser split, maps the common lexer constructs to Nest VPA rules, and maps ANTLR parser rules to Nest PEG rules. Unsupported target-language semantic actions are skipped because Nest grammars are declarative.

## Command line

```sh
g4 <input.g4>... -o <output.nest> [MODE=parserRule ...]
```

- One or more `input.g4` files are accepted. This supports ANTLR projects split into lexer and parser grammars.
  - All input files are parsed before generation.
  - A file whose grammar header starts with `lexer grammar` is a lexer grammar and contributes lexer rules, modes, fragments, token declarations, and channels.
  - If more than one `lexer grammar` file is supplied, the last `lexer grammar` file in command-line order is the lexer entrance grammar. Only this entrance lexer grammar emits Nest VPA scopes and token-producing lexer lines. Earlier `lexer grammar` files are support lexer grammars: their lexer rules are emitted only as `%define` regexp fragments for reuse by the entrance lexer grammar and do not create VPA scopes, mode entrances, `%ignore` entries, or token-producing lines by themselves.
  - A file whose grammar header starts with `parser grammar` is a parser grammar and contributes only parser rules and parser-side declarations.
  - A file whose grammar header starts with `grammar` is a combined grammar and may contribute both lexer and parser definitions.
  - Lexer rules, modes, fragments, token declarations, parser rules, and channels are processed in command-line order, subject to the rule that only the last `lexer grammar` file is the lexer entrance when multiple lexer grammars are present.
  - Duplicate grammar definitions are errors unless ANTLR permits the construct as a declaration-only reference.
- `-o output.nest`: generated Nest grammar.
- `MODE=parserRule`: required for every non-default ANTLR lexer mode that appears in the merged grammar unless that mode is explicitly mapped to an empty value.
  - `MODE` is the ANTLR mode name, for example `STRING`.
  - `parserRule` is the ANTLR parser rule that should parse tokens produced while that mode is active.
  - The parser rule name is converted to Nest snake case and becomes the Nest scope name for that mode.
  - `MODE=` means the importer must not create a Nest scope wrapper for that ANTLR mode. Rules in that mode are inlined/merged into the default lexer scope using their effective token actions. Use this for ANTLR modes such as Kotlin `Inside` whose tokens are parsed by different PEG rules depending on context, rather than by one uniform scope parser.
  - If any non-default mode has neither a `MODE=parserRule` mapping nor an explicit `MODE=` empty mapping, the command must print an error naming the unmapped mode and exit without writing output.
- `LEXER_RULE=parserRule`: required for every recursive lexer rule that is not sent to the hidden channel and is not skipped.
  - Recursive lexer rules are imported as Nest scopes.
  - If such a scope yields visible tokens, it must have a parser rule mapping so the resulting grammar can operate at the scope level.
  - Hidden or skipped recursive lexer scopes do not require a parser rule mapping.
- The ANTLR default mode maps to the generated Nest `*default_mode` macro. The generated Nest `main` scope is only an EOF wrapper around that macro:

  ```nest
  main = {
    *default_mode
    EOF .end
  }
  ```

  The importer must not generate a separate `default_mode` wrapper around `*default_mode`; that wrapper is semantically redundant and incorrectly mixes root EOF handling into nested mode transitions.

Example:

```sh
g4 JavaLexer.g4 JavaParser.g4 -o java.nest STRING=stringLiteral TEMPLATE=templateLiteral
```

## Name mapping

ANTLR names are normalized to names that are valid in Nest syntax.

- ANTLR lexer rule names start with an uppercase letter and map to `@`-prefixed Nest token names.
  - The emitted token name is the lexer rule name converted to snake case.
  - `IDENTIFIER` -> `@identifier`
  - `STRING_START` -> `@string_start`
  - `ShebangLine` -> `@shebang_line`
  - `LineStrExprStart` -> `@line_str_expr_start`
- Camel-cased parser rule names map to snake case Nest PEG rule names.
  - `compilationUnit` -> `compilation_unit`
  - `classBodyDeclaration` -> `class_body_declaration`
- The ANTLR `DEFAULT_MODE` maps to the generated Nest VPA macro `*default_mode`; Nest `main` wraps `*default_mode` and EOF.
- Non-default ANTLR modes map to Nest VPA scopes using the required command-line mode-to-parser-rule mapping.
- Recursive lexer rules map to Nest scopes named by their snake-case lexer rule name.

## Lexer mapping

ANTLR lexer rules become Nest VPA rules or scope lines.

```antlr
TOKEN : 'literal' ;
WS    : [ \t\r\n]+ -> channel(HIDDEN) ;
COMMENT : '//' ~[\r\n]* -> skip ;
```

becomes approximately:

```nest
[[vpa]]

%ignore @ws

main = {
  *default_mode
  EOF .end
}

*default_mode = {
  "literal" @token
  /[ \t\r\n]/+ @ws
  "//" /[^\r\n]/*
}
```

ANTLR lexer pattern actions are processed in this order:

1. **Collect the raw action.**
   - A `-> ...` lexer command list is an action.
   - A target-language semantic action `{ ... }` inside the lexer alternative is also an action.
   - If not exist, consider it an empty list
2. **Normalize the action before interpreting commands.**
   - If the command list contains no `type(...)` command, prepend `type(PATTERN_NAME)` (the owning ANTLR lexer rule name)
   - If the list (TODO: if..else should actually be hooks)
3. **Analyze if the emit token is used**
   1. Collect all used pattern names in parsers first (`pattern.used = true`)
   2. Patterm can reference other pattern, then the referenced ones should be marked (`pattern.referenced = true`)
   3. Only used pattern generates a line in the scope
   4. For referenced pattern,
      - if non-recursive, generate as `%define` pattern so that patterns using it can reference
      - if recursive, must map to scope (see handling below)
4. **Interpret supported commands left-to-right.** The importer supports comma-separated command lists after `->`.
   - `skip` suppresses token emission.
   - `type(TOKEN_NAME)` sets the effective emitted Nest token to `@snake_case(TOKEN_NAME)`.
   - `channel(HIDDEN)` records the effective emitted token for `%ignore`.
   - `pushMode(MODE)` and `mode(MODE)` create a Nest scope-enter action.
   - `popMode` maps to `.end`

ANTLR lexer fragments and reusable lexer-rule bodies become named Nest regexp fragments. The importer defines uppercase rule-reference ids first and references them from later lexer rules.

ANTLR lexer negation `~` maps to a Nest negative character class, not to a literal `~` regex operator. The importer flattens literal and charset alternatives inside the negation into one character class when possible.

- `~[\r\n]` -> `/[^\r\n]/`
- `~'"'` -> `/[^"]/`
- `~('a' | [bc])` -> `/[^abc]/`

```antlr
fragment DIGIT : [0-9] ;
INT : DIGIT+ ;
RETURN_AT : 'return@' Identifier ;
```

becomes approximately:

```nest
%define DIGIT /[0-9]/
%define IDENTIFIER /.../
%define INT DIGIT+
%define RETURN_AT "return@" IDENTIFIER

main = {
  *default_mode
  EOF .end
}

*default_mode = {
  INT @int
  RETURN_AT @return_at
}
```

Rules:

- A non-recursive `fragment` lexer rule defines an uppercase Nest regexp fragment and never emits a token by itself.
- A non-fragment lexer rule also defines an uppercase Nest regexp fragment for references from other lexer rules when its body is pure regular expression syntax.
- A recursive lexer rule cannot be represented as a pure regexp fragment. It is emitted as a Nest VPA scope, not as `%define`.
- Any lexer rule whose body references a recursive lexer rule, directly or through fragments, must also be emitted as a Nest scope because the reference can cross scope boundaries. This includes non-recursive rules such as `EXCL_WS: '!' Hidden;` when `Hidden` expands to a recursive `DelimitedComment`.
- If multiple `lexer grammar` files are supplied, non-fragment lexer rules from support lexer grammars before the last lexer grammar are treated like reusable definitions only: pure regular rules emit `%define` fragments, while scope-like rules may be used to build entrance scopes but do not emit tokens directly.
- A non-fragment lexer rule from the entrance lexer grammar, or from a combined grammar, additionally appears as a VPA line in its ANTLR mode unless it is emitted as a generated scope body.
- If a token-producing lexer rule has no manual `->` command/effect and its default token is not referenced by any imported parser rule, the rule is omitted from generated scopes. Omission is not the same as `skip`: a skipped rule still appears as a consuming non-token pattern, while an unused default-token rule does not appear at all. This fallback handles lexer/parser grammar mismatches without creating visible tokens that the generated PEG cannot consume.
- A lexer reference like `Identifier` inside a pure regexp rule is emitted as the uppercase fragment reference `IDENTIFIER`.

Example transitive recursive reference:

```antlr
EXCL_WS : '!' Hidden ;
fragment Hidden : DelimitedComment | LineComment | WS ;
DelimitedComment : '/*' (DelimitedComment | .)*? '*/' -> channel(HIDDEN) ;
LineComment : '//' ~[\r\n]* ;
WS : [ \t\f]+ ;
```

`DelimitedComment` is recursive, so `EXCL_WS` must be emitted as a scope instead of a single regexp. The fragment `Hidden` is expanded into scope alternatives:

```nest
excl_ws = "!" .begin {
  delimited_comment @excl_ws .end
  /\/\/[^\r\n]*/ @excl_ws .end
  /[ \t\f]+/ @excl_ws .end
}
```

## Mode and scope mapping

ANTLR modes map to Nest scopes. ANTLR `pushMode` / `popMode` maps to Nest's scope hierarchy.

- Mode-to-scope emission is driven by the entrance lexer grammar. When multiple `lexer grammar` files are supplied, modes declared only in support lexer grammars before the last lexer grammar are available only through `%define` fragments and do not create Nest scopes by themselves.
- A lexer rule with `-> pushMode(MODE)` begins the Nest scope mapped from `MODE`.
  - The rule's lexer pattern becomes the opening pattern for that scope.
  - The generated action is `.begin`, plus an emitted token when the rule is not skipped.
  - If multiple rules can push the same mode, each rule becomes an entry line to the same Nest scope.
- A lexer rule with `-> popMode` ends the current Nest scope.
  - The generated action is `.end`, plus an emitted token when the rule is not skipped.
- Nested ANTLR mode pushes therefore become nested Nest token chunks/scopes.
- `DEFAULT_MODE` is emitted as the Nest `*default_mode` macro.
- Nest `main` is generated as the only VPA scope that handles source EOF, wrapping `*default_mode` and EOF, for example:

```nest
main = {
  *default_mode
  EOF .end
}
```

- The default lexer body is emitted as `*default_mode = { ... }`; no `default_mode = /./ .unparse .begin { *default_mode EOF .end }` wrapper is emitted.
- A root-level `pushMode(DEFAULT_MODE)` therefore enters a generated edge wrapper for the transition, rather than trying to make `main` recursive.
- If a non-default mode is mapped with an empty command-line value (`MODE=`), the importer does not emit wrappers or a macro for that mode. Its token rules are emitted into the generated `*default_mode` macro (with their effective `type(...)` token names), and its `popMode` rules are emitted as ordinary token-producing rules rather than `.end` actions for a Nest scope.

```antlr
STRING_START : '"' -> pushMode(STRING) ;
mode STRING;
STRING_TEXT : ~["\\]+ ;
STRING_END  : '"' -> popMode ;
```

with `STRING=stringLiteral` becomes approximately:

```nest
main = {
  *default_mode
  EOF .end
}

*default_mode = {
  default_mode__string
}

*string = {
  /[^"\\]/+ @string_text
  "\"" @string_end .end
}

default_mode__string = /"/ .begin @string_start {
  *string
}
```

Every non-flat `pushMode` / `mode` transition is emitted as a caller/callee edge wrapper named `{caller_scope}__{callee_scope}` after snake-case conversion. For example `DEFAULT_MODE -> LineString` is `default_mode__line_string`, and `LineString -> DEFAULT_MODE` is `line_string__default_mode`. Edge wrappers contain the opening pattern/action and the callee macro only; they must not include `EOF .end`. EOF handling belongs only in `main`. Edge-specific command-line mappings such as `DEFAULT_MODE:LineString=lineStringLiteral` are accepted as parser-rule coverage declarations for validation; they must not rename the generated VPA edge wrapper. For every non-empty edge mapping `foo:bar=baz`, the importer also emits a PEG wrapper named with the same caller/callee rule name: `foo__bar = baz` after Nest snake-case conversion.

For example `DEFAULT_MODE:LineString=lineStringLiteral` emits `default_mode__line_string = line_string_literal` in `[[peg]]`. Parser references to the mapped parser rule are rewritten to call the edge wrapper instead; for example a default-mode PEG occurrence of `lineStringLiteral` is emitted as `default_mode__line_string`, so the PEG follows the same scoped token chunk that the VPA creates.

## Recursive lexer rules

A recursive lexer rule is a lexer rule whose body references itself, directly or through other lexer rules. Recursive lexer rules are imported as Nest scopes instead of pure regexp fragments.

```antlr
DelimitedComment
    : '/*' ( DelimitedComment | . )*? '*/'
      -> channel(HIDDEN)
    ;
```

becomes approximately:

```nest
delimited_comment = "/\\*" .begin @delimited_comment {
  "*/" .end
  delimited_comment
  /./
}
```

Recursive lexer scopes must preserve the greedy/non-greedy ordering of the repeated middle pattern relative to the closing pattern:

- `A X*? C`, `A X+? C`, and similar non-greedy forms prefer the close pattern first, then the recursive/body alternatives. They lower approximately to:

  ```nest
  scope = A .begin {
    C .end
    X
  }
  ```

- `A X* C`, `A X+ C`, and similar greedy forms prefer the recursive/body alternatives first, then the close pattern. They lower approximately to:

  ```nest
  scope = A .begin {
    X*
    C .end
  }
  ```

For recursive lexer rules, non-greedy multipliers are therefore not merely diagnostic: they determine whether the generated scope tests the close pattern before or after the body pattern.

- If the recursive rule is hidden (`channel(HIDDEN)`) or skipped, the generated scope does not require a parser rule mapping.
- If the recursive rule can emit visible tokens, the command line must include `LexerRuleName=parserRule`; otherwise the command must print an error naming the recursive rule and exit without writing output.
- Non-greedy lexer multipliers (`*?`, `+?`, `??`) outside this recursive-scope lowering are parsed and reported to the user with source locations.
- Until Nest multiplier handling is improved for non-recursive contexts, unsupported non-greedy multipliers are lowered with the closest generated Nest form and the output file is still written.
- The command must print a warning when generated output outside recursive-scope lowering contains non-greedy approximations and must be reviewed/edited manually after generation.

## Parser mapping

ANTLR parser rules become Nest PEG rules.

- Alternatives (`|`) become Nest branch blocks (`[ ... ]`).
- Parenthesized parser subexpressions create generated helper rules named `_{snake_case(rule_name)}_{paren_id}`.
  - `rule_name` is the containing ANTLR parser rule name after snake-case conversion.
  - `paren_id` is a 1-based counter assigned by encounter order within that declared parser rule.
  - The counter resets to 1 for each parser rule declaration; it is not a global counter across the grammar and is not shared with other declared rules.
  - The original parenthesized expression is replaced by a call to the generated helper rule, preserving any `?`, `*`, or `+` suffix on the call.
- Token references become `@token_name` terms.
- Parser references become snake-case rule calls.
- EBNF suffixes `?`, `*`, and `+` are preserved.
- Labels (`x=rule`, `x+=rule`) and ANTLR alternative labels (`#Name`) are dropped.
- `EOF` is dropped because Nest VPA scopes end at scope boundaries.
- Semantic actions and predicates are dropped.

```antlr
compilationUnit : statement* EOF ;
statement       : RETURN expression ';' | expression ';' ;
```

becomes:

```nest
[[peg]]

main = compilation_unit
compilation_unit = statement*
statement = [ @return expression ";" expression ";" ]
```

If the input grammar has no parser rule named `main`, the importer creates a `main = first_rule` entrance rule.

## Limits

The importer is a bootstrap/convenience tool, not a full ANTLR semantic clone. It deliberately ignores target-language actions and predicates. It supports lexer commands `skip`, `channel(HIDDEN)`, `type(...)`, `pushMode(...)`, `mode(...)`, and `popMode`. Grammar imports/options are accepted for source organization, but the converter relies on all required source files being supplied on the command line and reports unresolved references as errors. The generated Nest grammar should be reviewed when the source grammar uses ANTLR-specific semantic features.
