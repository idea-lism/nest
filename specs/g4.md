# ANTLR g4 importer

`src/g4.c` is a small source-to-source importer from ANTLR `.g4` grammars to Nest grammar files.

## Mandatory implementation architecture

This importer is a bootstrapping exercise for Nest itself. Recognition of `.g4` syntax **must** be performed by a Nest grammar compiled by `nest`.

Required implementation shape:

1. Create and maintain a dedicated Nest grammar for ANTLR `.g4` input, for example `src/g4.nest` or another checked-in `.nest` file chosen by the build.
2. Add build rules that compile that Nest grammar with the Nest compiler and link the generated lexer/parser artifacts into the `g4` executable.
3. Implement `src/g4.c` only as the driver/converter around the generated parser: command-line handling, invoking the generated parser, walking the parse tree/AST, diagnostics, semantic conversion, and Nest output generation.

Forbidden implementation shapes:

- Do **not** hand-write a `.g4` lexer or parser in C.
- Do **not** implement recognition as a regex-only scanner.
- Do **not** use ANTLR, yacc/bison, tree-sitter, or any other parser generator for `.g4` syntax.
- Do **not** check in a C file whose tokenization/parsing logic recognizes `.g4` directly instead of calling the Nest-generated parser.

If the generated Nest parser cannot yet recognize a `.g4` construct, extend the dedicated `.nest` grammar first. The checked-in C importer may contain command-line handling, AST conversion, diagnostics, and Nest output generation, but recognition of `.g4` syntax must come from the Nest-generated lexer/parser artifacts.

The importer is intentionally conservative: it preserves ANTLR's high-level lexer/parser split, maps the common lexer constructs to Nest VPA rules, and maps ANTLR parser rules to Nest PEG rules. Unsupported target-language semantic actions are skipped because Nest grammars are declarative.

## Command line

```sh
g4 <input.g4>... -o <output.nest> [MODE=parserRule ...]
```

- One or more `input.g4` files are accepted. This supports ANTLR projects split into lexer and parser grammars.
  - All input files are parsed before generation.
  - A file whose grammar header starts with `lexer grammar` is a lexer grammar and contributes only lexer rules, modes, fragments, token declarations, and channels.
  - A file whose grammar header starts with `parser grammar` is a parser grammar and contributes only parser rules and parser-side declarations.
  - A file whose grammar header starts with `grammar` is a combined grammar and may contribute both lexer and parser definitions.
  - Lexer rules, modes, fragments, token declarations, parser rules, and channels are merged in command-line order.
  - Duplicate grammar definitions are errors unless ANTLR permits the construct as a declaration-only reference.
- `-o output.nest`: generated Nest grammar.
- `MODE=parserRule`: required for every non-default ANTLR lexer mode that appears in the merged grammar.
  - `MODE` is the ANTLR mode name, for example `STRING`.
  - `parserRule` is the ANTLR parser rule that should parse tokens produced while that mode is active.
  - The parser rule name is converted to Nest snake case and becomes the Nest scope name for that mode.
  - If any non-default mode has no command-line mapping, the command must print an error naming the unmapped mode and exit without writing output.
- `LEXER_RULE=parserRule`: required for every recursive lexer rule that is not sent to the hidden channel and is not skipped.
  - Recursive lexer rules are imported as Nest scopes.
  - If such a scope yields visible tokens, it must have a parser rule mapping so the resulting grammar can operate at the scope level.
  - Hidden or skipped recursive lexer scopes do not require a parser rule mapping.
- The ANTLR default mode maps to the generated Nest `default` scope. The generated Nest `main` scope wraps `default` and EOF.

Example:

```sh
g4 JavaLexer.g4 JavaParser.g4 -o java.nest STRING=stringLiteral TEMPLATE=templateLiteral
```

## Name mapping

ANTLR names are normalized to names that are valid in Nest syntax.

- ANTLR lexer rule names start with an uppercase letter and map to `@`-prefixed Nest token names.
  - The emitted token name is the lexer rule name converted to snake case.
  - `IDENTIFIER` -> `@identifier`
  - `STRING_START` -> `@string_start`
  - `ShebangLine` -> `@shebang_line`
  - `LineStrExprStart` -> `@line_str_expr_start`
- Camel-cased parser rule names map to snake case Nest PEG rule names.
  - `compilationUnit` -> `compilation_unit`
  - `classBodyDeclaration` -> `class_body_declaration`
- The ANTLR `DEFAULT_MODE` maps to the generated Nest VPA scope `default`; Nest `main` wraps `default` and EOF.
- Non-default ANTLR modes map to Nest VPA scopes using the required command-line mode-to-parser-rule mapping.
- Recursive lexer rules map to Nest scopes named by their snake-case lexer rule name.

## Lexer mapping

ANTLR lexer rules become Nest VPA rules or scope lines.

```antlr
TOKEN : 'literal' ;
WS    : [ \t\r\n]+ -> channel(HIDDEN) ;
COMMENT : '//' ~[\r\n]* -> skip ;
```

becomes approximately:

```nest
[[vpa]]

%ignore @ws

main = {
  "literal" @token
  /[ \t\r\n]/+ @ws
  "//" /[^\r\n]/*
  EOF .end
}
```

Rules with `-> channel(HIDDEN)` are emitted as tokens and listed in `%ignore`. Rules with `-> skip` are emitted without a token action, so they consume input but do not yield a token.

ANTLR lexer commands are processed left-to-right. The importer supports comma-separated command lists after `->`.

- `type(TOKEN_NAME)` changes the emitted Nest token to `@snake_case(TOKEN_NAME)` instead of the current lexer rule's token name.
  - `Inside_RPAREN: RPAREN -> popMode, type(RPAREN);` becomes `RPAREN .end @rparen` and emits `@rparen`, not `@inside_rparen`.
  - `Inside_Identifier: Identifier -> type(Identifier);` emits `@identifier`.
- `channel(HIDDEN)` uses the effective emitted token name after `type(...)` when adding `%ignore` entries.
- `skip` suppresses token emission even if `type(...)` is also present.

ANTLR lexer fragments and reusable lexer-rule bodies become named Nest regexp fragments. The importer defines uppercase rule-reference ids first and references them from later lexer rules.

```antlr
fragment DIGIT : [0-9] ;
INT : DIGIT+ ;
RETURN_AT : 'return@' Identifier ;
```

becomes approximately:

```nest
%define DIGIT /[0-9]/
%define IDENTIFIER /.../
%define INT DIGIT+
%define RETURN_AT "return@" IDENTIFIER

default = {
  INT @int
  RETURN_AT @return_at
  EOF .end
}
```

Rules:

- A `fragment` lexer rule defines an uppercase Nest regexp fragment and never emits a token by itself.
- A non-fragment lexer rule also defines an uppercase Nest regexp fragment for references from other lexer rules.
- A non-fragment lexer rule additionally appears as a VPA line in its ANTLR mode unless it is used only as a generated recursive scope body.
- A lexer reference like `Identifier` inside another lexer rule is emitted as the uppercase fragment reference `IDENTIFIER`.

## Mode and scope mapping

ANTLR modes map to Nest scopes. ANTLR `pushMode` / `popMode` maps to Nest's scope hierarchy.

- A lexer rule with `-> pushMode(MODE)` begins the Nest scope mapped from `MODE`.
  - The rule's lexer pattern becomes the opening pattern for that scope.
  - The generated action is `.begin`, plus an emitted token when the rule is not skipped.
  - If multiple rules can push the same mode, each rule becomes an entry line to the same Nest scope.
- A lexer rule with `-> popMode` ends the current Nest scope.
  - The generated action is `.end`, plus an emitted token when the rule is not skipped.
- Nested ANTLR mode pushes therefore become nested Nest token chunks/scopes.
- `DEFAULT_MODE` is emitted as the Nest `default` scope.
- Nest `main` is generated as a wrapper around `default` and EOF, for example:

```nest
main = {
  default
  EOF .end
}
```

- A root-level `pushMode(DEFAULT_MODE)` therefore enters the `default` scope, rather than trying to make `main` recursive.

```antlr
STRING_START : '"' -> pushMode(STRING) ;
mode STRING;
STRING_TEXT : ~["\\]+ ;
STRING_END  : '"' -> popMode ;
```

with `STRING=stringLiteral` becomes approximately:

```nest
main = {
  string_literal
  EOF .end
}

string_literal = "\"" @string_start .begin {
  /[^"\\]/+ @string_text
  "\"" @string_end .end
}
```

`pushMode(MODE)` is the preferred ANTLR command because it maps directly to Nest's visibly-pushdown scope stack. `mode(MODE)` is imported as a scope enter as well; grammars that rely on replacement-mode semantics may need manual cleanup.

## Recursive lexer rules

A recursive lexer rule is a lexer rule whose body references itself, directly or through other lexer rules. Recursive lexer rules are imported as Nest scopes instead of pure regexp fragments.

```antlr
DelimitedComment
    : '/*' ( DelimitedComment | . )*? '*/'
      -> channel(HIDDEN)
    ;
```

becomes approximately:

```nest
delimited_comment = "/\\*" .begin @delimited_comment {
  delimited_comment
  "*/" .end
  /./
}
```

- If the recursive rule is hidden (`channel(HIDDEN)`) or skipped, the generated scope does not require a parser rule mapping.
- If the recursive rule can emit visible tokens, the command line must include `LexerRuleName=parserRule`; otherwise the command must print an error naming the recursive rule and exit without writing output.
- Non-greedy lexer multipliers (`*?`, `+?`, `??`) are parsed and reported to the user with source locations.
- Until Nest multiplier handling is improved, non-greedy multipliers are lowered with the closest generated Nest form and the output file is still written.
- The command must print a warning that the generated output contains non-greedy approximations and must be reviewed/edited manually after generation.

## Parser mapping

ANTLR parser rules become Nest PEG rules.

- Alternatives (`|`) become Nest branch blocks (`[ ... ]`).
- Parenthesized parser subexpressions create generated helper rules named `_{snake_case(rule_name)}_{paren_id}`.
  - `rule_name` is the containing ANTLR parser rule name after snake-case conversion.
  - `paren_id` is a 1-based counter assigned by encounter order within that containing parser rule.
  - The original parenthesized expression is replaced by a call to the generated helper rule, preserving any `?`, `*`, or `+` suffix on the call.
- Token references become `@token_name` terms.
- Parser references become snake-case rule calls.
- EBNF suffixes `?`, `*`, and `+` are preserved.
- Labels (`x=rule`, `x+=rule`) and ANTLR alternative labels (`#Name`) are dropped.
- `EOF` is dropped because Nest VPA scopes end at scope boundaries.
- Semantic actions and predicates are dropped.

```antlr
compilationUnit : statement* EOF ;
statement       : RETURN expression ';' | expression ';' ;
```

becomes:

```nest
[[peg]]

main = compilation_unit
compilation_unit = statement*
statement = [ @return expression ";" expression ";" ]
```

If the input grammar has no parser rule named `main`, the importer creates a `main = first_rule` entrance rule.

## Limits

The importer is a bootstrap/convenience tool, not a full ANTLR semantic clone. It deliberately ignores target-language actions and predicates. It supports lexer commands `skip`, `channel(HIDDEN)`, `type(...)`, `pushMode(...)`, `mode(...)`, and `popMode`. Grammar imports/options are accepted for source organization, but the converter relies on all required source files being supplied on the command line and reports unresolved references as errors. The generated Nest grammar should be reviewed when the source grammar uses ANTLR-specific semantic features.
